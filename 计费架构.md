# 积分计费技术架构设计 v1

> 给现有「赛道智能体操盘台」加上 **账号 + 积分钱包 + 扣费 + 支付**，把无状态代理升级成可收费的服务。
> 设计目标：**钱不丢、不超扣、不白嫖、可对账、易扩展**。

---

## 0. 现状与改造点
- 现状：`server.js` 是**无状态代理**，前端直接调 `/api/claude`、`/api/image` 等，**无登录、无计费**。
- 改造：在所有 `/api/*` 业务接口前加一层 **鉴权 + 计费中间件**；新增 `账号 / 钱包 / 订单 / 支付` 模块与数据库。

```
浏览器（带 JWT）
   │
   ▼
[鉴权中间件] ── 无 token → 401
   │
[计费中间件] ── 余额不足 → 402（前端弹充值）
   │  1) 预扣（hold）  2) 调 AI  3) 成功结算 / 失败回滚
   ▼
现有 /api/claude · /api/image · /api/vision …
```

---

## 1. 技术选型（建议）
| 关注点 | 选型 | 理由 |
|---|---|---|
| 服务端 | Node 18+（沿用现有 server.js，或迁 Express/Fastify） | 与现有一致 |
| 数据库 | **PostgreSQL**（事务 + `SELECT … FOR UPDATE` + 唯一约束） | 钱必须强一致 |
| 缓存/锁/幂等 | **Redis**（分布式锁、幂等键、限流） | 高并发扣费 |
| 鉴权 | **JWT**（手机号验证码 / 微信登录换取） | 无状态、易扩展 |
| 支付 | **微信支付 Native(扫码)** + **支付宝当面付/电脑网站支付** | 国内主流 |

> 起步可只用 Postgres（事务足够），并发量上来再引 Redis。

## 2. 数据表设计（核心 5 张）

```sql
-- 用户
CREATE TABLE users (
  id           BIGSERIAL PRIMARY KEY,
  phone        VARCHAR(20) UNIQUE,          -- 手机号登录
  wx_openid    VARCHAR(64) UNIQUE,          -- 微信登录（二选一/都可）
  nickname     VARCHAR(64),
  status       SMALLINT DEFAULT 1,          -- 1正常 0封禁
  created_at   TIMESTAMPTZ DEFAULT now()
);

-- 钱包（缓存当前余额，真值以流水为准）
CREATE TABLE wallets (
  user_id      BIGINT PRIMARY KEY REFERENCES users(id),
  balance      INTEGER NOT NULL DEFAULT 0,  -- 当前可用积分
  frozen       INTEGER NOT NULL DEFAULT 0,  -- 预扣冻结中
  updated_at   TIMESTAMPTZ DEFAULT now(),
  CHECK (balance >= 0 AND frozen >= 0)
);

-- 积分流水（不可变，对账/审计源头）
CREATE TABLE credit_ledger (
  id           BIGSERIAL PRIMARY KEY,
  user_id      BIGINT NOT NULL REFERENCES users(id),
  type         VARCHAR(16) NOT NULL,        -- grant赠送 recharge充值 hold预扣 settle结算 refund回滚 consume直扣
  amount       INTEGER NOT NULL,            -- 正=入账 负=出账
  balance_after INTEGER NOT NULL,
  ref_type     VARCHAR(24),                 -- 'order' | 'usage' | 'signup'…
  ref_id       VARCHAR(64),                 -- 订单号 / 用量请求号
  request_id   VARCHAR(64) UNIQUE,          -- 幂等键：同一请求只扣一次
  meta         JSONB,                       -- {action:'image', n:6, model:'cogview-3-flash'}
  created_at   TIMESTAMPTZ DEFAULT now()
);

-- 计价规则（可后台改，不写死在代码）
CREATE TABLE price_rules (
  action_key   VARCHAR(32) PRIMARY KEY,     -- 'topic'|'note_text'|'image_std'|'image_hd'|'compliance'…
  credits      INTEGER NOT NULL,
  active        BOOLEAN DEFAULT true
);

-- 充值订单
CREATE TABLE orders (
  id           BIGSERIAL PRIMARY KEY,
  out_trade_no VARCHAR(64) UNIQUE NOT NULL, -- 我方订单号（给支付渠道）
  user_id      BIGINT NOT NULL REFERENCES users(id),
  pack_id      VARCHAR(32),                 -- 套餐ID
  amount_cny   INTEGER NOT NULL,            -- 实付金额（分）
  credits      INTEGER NOT NULL,            -- 应到账积分（含赠送）
  status       VARCHAR(16) DEFAULT 'pending', -- pending|paid|failed|refunded
  pay_channel  VARCHAR(16),                 -- wxpay|alipay
  transaction_id VARCHAR(64),               -- 渠道流水号
  created_at   TIMESTAMPTZ DEFAULT now(),
  paid_at      TIMESTAMPTZ
);
```

> **真值是 `credit_ledger`**，`wallets.balance` 是为查询加速的物化值，二者在同一事务内原子更新。

## 3. 扣费流程（预扣 + 结算/回滚，保证不超扣、失败不收费）

每次业务调用（如生成 N 张图）的请求带一个 **`request_id`（前端 UUID）** 做幂等。

```
1) 算价：cost = Σ price_rules（按本次动作，如 image_std × N）
2) 预扣(hold)：一个事务里
     SELECT balance FROM wallets WHERE user_id=? FOR UPDATE;   -- 行锁
     若 balance < cost → 回滚，返回 402 余额不足
     UPDATE wallets SET balance=balance-cost, frozen=frozen+cost;
     INSERT credit_ledger(type='hold', amount=-cost, request_id, ...);
3) 调用 AI（耗时操作，不在事务里）
4a) 成功 → settle：
     UPDATE wallets SET frozen=frozen-cost;                    -- 真正消费
     INSERT credit_ledger(type='settle', amount=0, ref='usage', meta=...);
4b) 失败 → refund：
     UPDATE wallets SET balance=balance+cost, frozen=frozen-cost;
     INSERT credit_ledger(type='refund', amount=+cost, ...);
```

要点：
- **行锁 `FOR UPDATE`** 防并发超扣；
- **`request_id` 唯一约束** 防前端重试/网络抖动导致双扣；
- 失败（AI 报错/超时）**全额回滚**，用户不为失败买单；
- 部分成功（如 6 张图成 4 张）：按**实际成功张数结算**，其余回滚。

> 简化版（小并发可用）：先 `SELECT balance` 校验 → 调 AI → 成功后 `UPDATE … WHERE balance>=cost` 原子直扣。代价是极端并发下可能"已出图但扣费失败"（罕见，金额小，可接受）。**正式建议用预扣方案。**

## 4. 充值 / 支付流程（钱只从「服务端回调」入账）

```
下单：前端选套餐 → POST /api/order/create
      服务端建 orders(status=pending, out_trade_no) → 调微信/支付宝「统一下单」→ 返回二维码/支付链接
支付：用户扫码付款
回调：渠道异步 POST 我方 /api/pay/notify
      → 验签（必须！）→ 查 out_trade_no → 若 status!=paid：
          一个事务：orders.status=paid + 钱包加积分 + credit_ledger(type='recharge')
      → 幂等：同一 out_trade_no 重复回调只入账一次
查询：前端轮询 /api/order/status（兜底，防回调丢失）
```

铁律：
- **积分入账只发生在「验签通过的支付回调」里**，前端永远不能直接加积分；
- 回调**幂等**（`orders.out_trade_no` + `status` 判断）；
- 对账：每日核对 `orders(paid)` 总额 与 `credit_ledger(recharge)` 总额。

## 5. API 改造清单

新增（账号/钱包/支付）：
| 方法/路径 | 作用 |
|---|---|
| `POST /api/auth/send-code` | 发手机验证码 |
| `POST /api/auth/login` | 验证码/微信登录 → 返回 JWT |
| `GET  /api/wallet` | 余额 + 近期流水 |
| `GET  /api/price` | 当前计价规则（前端展示「本次约扣 X 积分」） |
| `POST /api/order/create` | 下单，返回支付二维码 |
| `POST /api/pay/notify` | 支付回调（验签 → 入账） |
| `GET  /api/order/status` | 订单状态轮询 |

改造（给现有业务接口加中间件）：
- `/api/claude`（按动作：`topic` / `note_text` / `compliance`）
- `/api/image`（`image_std` × N 或 `image_hd` × N）
- `/api/vision`（免费或并入文案）
- 统一：**未登录→401；余额不足→402**；前端拦 401 弹登录、拦 402 弹充值。

> 前端每个「生成」按钮旁显示 **「约扣 X 积分（余额 Y）」**，点完实时刷新余额。

## 6. 安全与风控
- **鉴权**：所有业务接口校验 JWT；JWT 短期 + refresh。
- **幂等**：`request_id`（用量）、`out_trade_no`（支付）唯一约束。
- **并发**：钱包行锁 / 原子条件更新，杜绝超扣。
- **限流**：按用户 + IP（Redis 滑动窗口），防刷接口烧成本。
- **防薅羊毛**：注册送积分绑手机/微信 + 同设备/IP 限领 + 风控名单。
- **审计**：`credit_ledger` 不可变，可完整回放每一分积分来龙去脉。
- **Key 仍只在服务端**（沿用现状）；新增 DB/支付密钥同样只在 `.env`。

## 7. 分阶段落地（建议顺序）
1. **MVP（能收钱）**：手机号登录 + 钱包 + 简化「后扣」+ 微信 Native 充值 + 注册送 200 积分。
2. **稳健版**：改预扣+回滚、加 Redis 幂等/限流、订单对账、支付宝。
3. **B 端**：多账号/团队、API Key 取稿、用量报表、OEM 配置（白牌）。
4. **增值**：知识库清洗工单、定制赛道下单、对标数据 API 接入。
